# Sprite Forge Studio — 이모티콘에서 게임 스프라이트로

Interactive **demo**: our blog-comic characters, first turned into KakaoTalk emoticons,
now extended into **game sprites**. Pick the robot, play its actions
(idle / run / jump / punch / **하도우켄·파동권** / victory), inspect the transparent
sprite sheet, and download a real Unity bundle.

**Live:** https://sylvanus4.github.io/sprite-forge-studio/

## Real vs simulated

- **Real:** every sprite is generated by the GPT Image 2 held-cel pipeline
  (`sticker-forge`) — the same one used for the emoticon set — then split, keyed to
  transparent, stabilized, and repacked into a uniform-cell sheet. The sprite sheet PNG,
  the Unity auto-slice `.png.meta`, `frames.json`, and the `.zip` bundle are all produced
  client-side and import into Unity.
- **Simulated:** only the on-screen "generation" progress animation.
- **WIP:** the robot is the finished game character. 박사(scientist) and 개발자(developer)
  are still going through the final game pass and are shown locked.

## Stack

100% client-side static site, zero dependencies, zero build step, system fonts,
dark-first with a `prefers-color-scheme` light fallback (sylvanus4 web-tool house style).

```
index.html            markup + panels
assets/style.css       design tokens
assets/app.js          loads real sprites, preview loop, Unity .meta/frames.json + store-zip
data/game.json         action manifest (webp + transparent sheet per action)
assets/game/robot/     real generated sprites: webp (animation) + sheets (transparent PNG)
```

## Behind it

The characters come from the ThakiCloud tech-blog comic. The held-cel approach (hold the
body, move one part) is what removes the frame-to-frame jitter; a stabilization pass then
locks foot baseline and size. Extending it from emoticon expressions to game actions
(run/jump/punch/파동권) reuses the exact same pipeline.
